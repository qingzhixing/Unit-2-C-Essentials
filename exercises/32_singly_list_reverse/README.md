## Lesson 32 — 单链表反转与快慢指针环检测【面试高频】

### 课程任务

实现技术面试中 C 语言手写代码的两道最高频题：

1. **`reverse(head)`** — 原地反转单链表：将 `1→2→3→NULL` 变为 `3→2→1→NULL`，返回新 head
2. **`has_cycle(head)`** — Floyd 快慢指针检测链表是否有环：有环返回 1，无环返回 0

`make_node`、`build_list`、`list_print`、`list_free` 已由模板提供。

### 为什么链表反转是面试高频？

链表反转在面试中出镜率极高，并非因为它难（它比红黑树简单得多），而是因为它在 15 分钟白板时间内可以高效考察候选人的多个维度：

| 面试官考察点 | 具体表现 | 评价标准 |
|-------------|---------|---------|
| **指针思维** | 能否在脑中建模 `prev/curr/next` 三指针的联动关系 | 指针是 C 语言的灵魂，链表操作直接反映指针功底 |
| **边界意识** | 空链表、单节点链表、双节点链表是否都考虑 | 面试官故意不说边界，看你是否主动询问或处理 |
| **代码整洁度** | 变量命名是否清晰（prev vs p），逻辑是否一目了然 | 白板代码就是你的代码风格名片 |
| **迭代 vs 递归** | 能否比较两种写法，并解释为什么迭代更优 | 展现对时间和空间复杂度的权衡意识 |
| **举一反三** | 被追问"反转部分链表""K个一组反转"时能否快速推导 | 区分"背过答案"和"真正理解"的关键分水岭 |

> **面试真相**：80% 的候选人能写出大致框架，但只有 20% 能正确处理所有边界条件。如果你在反转后能主动讨论"为什么返回 prev 而不是 curr"、"prev 初始值为什么是 NULL"，你已经进入了前 20%。

### 反转 `reverse(head)`：三指针法

#### 为什么需要三个指针？

反转 `[1]→[2]→[3]→NULL` 意味着把每个节点的 `next` 指向**反过来**。当你在改 `curr->next` 时，原来的 `curr->next` 信息会丢失——所以需要第三个指针 `next` 来提前保存。三个指针各司其职：

```
prev: 已经反转好的部分的头节点（初始 NULL，因为反转后原头节点的 next 指向 NULL）
curr: 当前正在处理的节点（初始 head）
next: 下一个要处理的节点（防止改 curr->next 后丢失剩余链表）
```

#### 完整步骤图解（带指针标签）

```
初始状态:
NULL    [1]→[2]→[3]→NULL
 ↑       ↑
prev    curr

═══════════════════════════════════
第 1 轮
═══════════════════════════════════

步骤 1: 保存 next = curr->next
NULL    [1]→[2]→[3]→NULL
 ↑       ↑    ↑
prev    curr next
// 不保存就丢失！

步骤 2: 反转 curr->next = prev
NULL ← [1]  [2]→[3]→NULL
 ↑       ↑    ↑
prev    curr next

步骤 3: prev = curr
NULL ← [1]  [2]→[3]→NULL
         ↑    ↑
    prev/curr next

步骤 4: curr = next
NULL ← [1]  [2]→[3]→NULL
         ↑    ↑
        prev curr

═══════════════════════════════════
第 2 轮（重复上述四步）
═══════════════════════════════════

next = curr->next → curr->next = prev → prev = curr → curr = next

NULL ← [1] ← [2]  [3]→NULL
              ↑     ↑
            prev   curr

═══════════════════════════════════
第 3 轮
═══════════════════════════════════

NULL ← [1] ← [2] ← [3]
                    ↑     curr = NULL
                   prev

循环结束（curr == NULL），返回 prev → 新 head = 节点 3 ✓
```

#### 关键参数速查

| 要素 | 值 | 说明 |
|------|-----|------|
| prev 初始值 | `NULL` | 反转后原头节点的 next 指向 NULL |
| curr 初始值 | `head` | 从头节点开始处理 |
| 循环条件 | `curr != NULL` | 处理完所有节点 |
| 返回值 | `prev` | curr 为 NULL 时 prev 是最后有效节点 |
| 时间复杂度 | O(n) | 每个节点访问一次 |
| 空间复杂度 | O(1) | 只用了三个指针，无额外分配 |

#### 递归 vs 迭代对比

| 维度 | 迭代（三指针法） | 递归 |
|------|----------------|------|
| 空间复杂度 | O(1) | O(n) — 递归调用栈 |
| 代码行数 | 5-7 行 | 3-5 行 |
| 栈溢出风险 | 无 | n > 10000 时危险 |
| 可读性 | 清晰直观 | 需理解递归返回语义 |
| 面试推荐 | ✅ 首选 | 可提及作为备选，但不如迭代稳妥 |
| 延伸变体 | 部分反转、K组反转 | 难以处理部分反转 |

#### 递归写法参考（了解即可）

```
reverse_recur(head):
  if (head == NULL || head->next == NULL) return head;  // 终止
  new_head = reverse_recur(head->next);                  // 递归到底
  head->next->next = head;                               // 反转
  head->next = NULL;                                     // 尾部置 NULL
  return new_head;
```

### 环检测 `has_cycle(head)`：Floyd 快慢指针

#### 核心原理

两个指针从 head 同时出发：`fast` 每轮走 2 步，`slow` 每轮走 1 步。

```
有环: fast 必然在环中追上 slow（操场套圈原理）
无环: fast 先到达 NULL（或 fast->next == NULL）
```

#### 为什么必相遇？（数学证明）

```
slow 和 fast 都进入环后:
  fast 相对于 slow 的速度 = 1 步 / 轮（fast 走 2 步, slow 走 1 步）
  
环的长度为 L，两者进入环时最多相距 L-1 步。
相对速度 1 步/轮，最多 L-1 轮后必然相遇。

关键: 步差必须为 1！
若 fast 走 3 步（步差 2），可能"跳过" slow 而不相遇。
所以 Floyd 算法的精髓是: slow 走 1, fast 走 2 → 步差 = 1 ✓
```

#### 逐步图解

```
有环链表: [1]→[2]→[3]→[4]
                    ↑   ↓
                   [6]←[5]

轮次 | slow 位置 | fast 位置 | 说明
0    | [1]      | [1]      | 起点
1    | [2]      | [3]      | 
2    | [3]      | [5]      | 
3    | [4]      | [3]      | fast 已绕环一周
4    | [5]      | [5]      | ← slow==fast 相遇！有环 ✓

无环链表: [1]→[2]→[3]→NULL

轮次 | slow 位置 | fast 位置 | 说明
0    | [1]      | [1]      | 起点
1    | [2]      | [3]      |
2    | [3]      | NULL     | ← fast 到 NULL, 无环 ✓
```

#### 循环条件详解

```c
while (fast != NULL && fast->next != NULL) {
    // 为什么必须同时检查两个条件？
    // fast 每次走 2 步 → 需要 fast 和 fast->next 都有效
    // fast->next->next 访问前，fast->next 不能为 NULL
}
```

#### 环入口查找（扩展知识）

```
相遇后: slow 回 head, fast 留相遇点
两者同速(每轮 1 步)前进 → 再次相遇处 = 环入口

数学证明:
  设 head→入口 距离 a, 环长 r, 相遇点距入口 s
  相遇时: slow 走了 a+s+(m·r), fast 走了 a+s+(n·r) = 2(a+s+(m·r))
  推导得 a = (n-2m-1)·r + (r-s)
  所以从 head 走 a 步 = 从相遇点走 (r-s) 步 → 两者在入口相遇
```

### 常见错误与陷阱

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 反转忘记保存 `next` | 链断，丢失剩余节点 | 三步顺序不可乱：保存→反转→前进 |
| 反转后返回 `curr` | 返回 NULL | 返回 `prev`（curr==NULL 时 prev 是最后节点） |
| 反转空链表不处理 | `curr->next` 段错误 | `if (head == NULL) return NULL;` |
| 环检测只判 `fast != NULL` | 访问 `fast->next->next` 时段错误 | 同时判断 `fast && fast->next` |
| 快慢指针初始不同步 | 可能漏判 | 都从 head 出发 |
| 环检测忘记单节点无环情况 | 单节点无环时 `fast->next` 为 NULL，应返回 0 | 循环条件自动处理 |

### 重要知识点

- 三指针法精髓：**保存 next → 反转 curr→next → 三指针前进**，三步顺序不可乱
- Floyd 算法数学保证：步差为 1 确保必相遇
- 反转是原地操作，不分配新节点——只修改 `next` 指针方向
- 环检测可扩展：找环入口（slow 回 head 同速前进）、计算环长

### 课堂讨论

1. **反转链表有递归写法吗？** — 有。递归到最后一个节点后逐层返回并反转。空间 O(n) 调用栈 vs 迭代 O(1)。迭代在面试中更稳妥。

2. **快指针走 3 步行不行？** — 不行。步差为 2 时可能"跳过去"而不相遇。必须步差 = 1。

3. **快慢指针还能用来做什么？** — 找链表中点（fast 到末尾时 slow 在中点）、找倒数第 K 个节点（fast 先走 K 步）、判断回文链表（找中点+反转后半+比较）。

4. **如何找到环的入口节点？** — 相遇后 slow 回 head，fast 留相遇点，同速前进直到再次相遇。

5. **链表反转有什么实际应用？** — undo/redo 操作栈、浏览器前进后退（双向链表简化版）、LRU 缓存中调整访问顺序。
