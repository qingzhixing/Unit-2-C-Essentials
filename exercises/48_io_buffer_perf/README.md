### 前置知识：为什么 I/O 有"缓冲"？

计算机科学中最慢的操作之一是磁盘 I/O——机械硬盘寻道 ~10ms，即使 SSD 也有 ~0.1ms 延迟。相比之下，内存访问约 100ns。**每次磁盘读写都是一次"系统调用"**（syscall），从用户态切换到内核态的开销是微秒级的。

```
应用程序
  ↓ fgetc/fputc (每次 1 字节)
用户态 ─────────────────────
内核态   ↓ 系统调用 (read/write)
  ↓ 磁盘I/O (极慢!)
─────── 硬件 ──────────────
```

**缓冲的本质**：在用户态攒够一批数据，一次性发给内核，减少系统调用次数。

```
无缓冲 (fgetc):  每个字节一次系统调用 → 1000 万次
有缓冲 (fread):  每 4096 字节一次系统调用 → 约 2500 次
减少 4000 倍!
```

### 三种缓冲模式

`FILE*` 支持三种缓冲模式，由 `setvbuf` 控制：

| 模式 | 行为 | 示例 | 默认场景 |
|------|------|------|---------|
| **全缓冲** (IOFBF) | 缓冲区满或显式 `fflush` 才写 | `fread`/`fwrite` 文件 | 磁盘文件 |
| **行缓冲** (IOLBF) | 遇 `\n` 或缓冲区满时写 | `printf` 到终端 | stdout (连终端时) |
| **无缓冲** (IONBF) | 立即写出 | `fprintf(stderr, ...)` | stderr |

```
全缓冲示例 (默认 FILE* 打开的文件):
  fwrite(buf, 1, 4096, f);  → 填满 4096 字节缓冲区 → 一次 write() 系统调用

行缓冲示例:
  printf("hello\n");  → 遇 \n → 立即 write() 到 stdout

无缓冲示例:
  fprintf(stderr, "error!");  → 立即 write()，不等待
```

### 算法详解

#### copy_fgetc — 逐字节拷贝

```
打开 src (读) 和 dst (写)
循环: fgetc 读 1 字节 → fputc 写 1 字节 → 重复 FILE_SIZE 次
关闭文件

每对 fgetc/fputc = 1+1 = 2 次系统调用
10MB 文件 = 10,485,760 字节 = 约 2000 万次系统调用!
```

#### copy_fread — 缓冲拷贝

```
打开 src 和 dst
循环: fread 读最多 4096 字节 → fwrite 写实际读到的字节数 → 重复
关闭文件

每次 fread/fwrite = 1+1 = 2 次系统调用
10MB / 4096 ≈ 2560 轮 → 约 5120 次系统调用
vs copy_fgetc 的 2000 万次 → 减少约 4000 倍!
```

### 逐步跟踪：10MB 文件拷贝

```
═══════════════════════════════════════════════
copy_fgetc 方式:
═══════════════════════════════════════════════
  fopen(src,"rb") → 返回 FILE* fin
  fopen(dst,"wb") → 返回 FILE* fout

  第 1 次循环: c = fgetc(fin) = 'A'  (1 次 read 系统调用, 但触发内核预读)
              fputc(c, fout)          (1 次 write 系统调用)
  第 2 次循环: 同上, 读 'B'
  ...
  第 10,485,760 次循环: 读最后 1 字节

  fclose(fin), fclose(fout)
  耗时: ~0.04 秒 (实际时间取决于磁盘速度)

═══════════════════════════════════════════════
copy_fread 方式:
═══════════════════════════════════════════════
  fopen(src,"rb"), fopen(dst,"wb")

  第 1 轮: n = fread(buf, 1, 4096, fin)  → n = 4096
          fwrite(buf, 1, 4096, fout)     → 写入这 4096 字节
  第 2 轮: 同上
  ...
  第 2560 轮: fread 读最后剩余字节, fwrite 写入

  fclose(fin), fclose(fout)
  耗时: ~0.005 秒 → 快了约 8 倍!
```

### 为什么 stdio 默认是全缓冲？

```
无缓冲 (每字节 syscall):
  ┌─usr─┬─sys─┬─disk─┬─sys─┬─usr─┬─sys─┬─disk─┬─...
  │app │kernel│ wait │kernel│app │kernel│ wait │
  └────┴─────┴──────┴─────┴────┴─────┴──────┴──
  大量时间花在用户态/内核态切换

全缓冲 (批量 syscall):
  ┌─usr (fill buffer)──────────┬─sys─┬─disk─┬─sys─┬─usr (fill)──...
  │app reads 4096 bytes to buf │kernel│ wait │kernel│app           │
  └────────────────────────────┴─────┴──────┴─────┴──────────────
  系统调用次数大幅减少 → 性能大幅提升
```

### 系统调用次数对比

| 方式 | 10MB 文件 | 系统调用次数 | 说明 |
|------|----------|------------|------|
| fgetc/fputc | 10,485,760 字节 | ~20,000,000 次 | 每字节 2 次 syscall |
| fread/fwrite (4KB) | 10,485,760 字节 | ~5,120 次 | 每 4KB 2 次 syscall |
| 理论加速比 | — | ~3906× | 实际受磁盘速度限制 |

实际加速比通常在 5-100× 之间，而非理论 4000×，因为磁盘物理读取速度成为瓶颈。但差异依然巨大。

### 常见错误

| 错误 | 后果 | 正确 |
|------|------|---------|
| `fread` 后忘记用返回值 | 写入脏数据 | `n = fread(...); fwrite(..., n)` |
| `fwrite` 用 sizeof(buf) 而非 n | 写入多余数据 | 用 `n`（实际读到的字节数） |
| 忘 `fclose` | 缓冲区未刷新，数据丢失 | 每个 fopen 对应一个 fclose |
| `fgetc` 返回 int，存为 char | EOF(-1) 与 0xFF 混淆 | `int c = fgetc(f)` |

### 重要知识点

- I/O 缓冲的本质：用批量操作减少系统调用次数
- 全缓冲（磁盘文件）vs 行缓冲（终端 stdout）vs 无缓冲（stderr）
- `fread/fwrite` 返回值是**实际读写的字节数**——必须使用它
- 10MB 逐字节拷贝约 2000 万次系统调用 → 极慢；4KB 缓冲仅 ~5000 次 → 快两个数量级

### 课堂讨论

1. **为什么 stderr 是无缓冲的？** — 错误信息需要立即输出。如果缓冲，程序崩溃时缓冲区内容丢失，看不到错误信息。

2. **如果缓冲区大小设为 1 字节，和 fgetc 有区别吗？** — 几乎无区别。缓冲区为 1 字节时，每次读写仍触发系统调用。

3. **为什么实际加速比远小于理论值？** — 理论值只看系统调用次数。但磁盘物理读取（寻道+旋转延迟+传输）是固定开销，4096 字节和 1 字节的磁盘操作时间差异不大。

4. **`fflush(fp)` 什么时候需要手动调用？** — 在需要保证数据已写入磁盘的场景（如日志落盘、数据库事务提交）。通常 fclose 自动调用。

5. **`setvbuf` 怎么用？** — `setvbuf(fp, buf, _IOFBF, 8192)` 可自定义缓冲区大小和模式。必须在任何 I/O 操作之前调用。

### setvbuf 自定义缓冲区

```c
#include <stdio.h>

char mybuf[8192];
FILE *f = fopen("data.bin", "rb");

/* 全缓冲: 使用自定义 8KB 缓冲区 */
setvbuf(f, mybuf, _IOFBF, sizeof(mybuf));

/* _IOFBF: 全缓冲  _IOLBF: 行缓冲  _IONBF: 无缓冲 */

/* 注意事项:
   - setvbuf 必须在任何 I/O 操作之前调用
   - 自定义缓冲区在 fclose 之前必须保持有效
   - 栈上的缓冲区在函数返回后失效 → 段错误!
*/
```

### 内核预读 (Readahead) 的影响

```
为什么 fgetc 的实际性能没有理论那么差？

内核的预读机制:
  当 fgetc 读取文件第 1 字节时:
    内核并非只读 1 字节 → 它预读了 128KB (默认值)!
    所以后续 127,999 次 fgetc 直接从内核缓存返回

  但每次 fgetc 仍触发 syscall (用户态↔内核态切换)
  这个上下文切换开销累积起来就是 fgetc 慢的根因

fread 的优势:
  一次 syscall 读取 4096 字节 → 4096 次用户态访问
  syscall 次数 = fgetc 的 1/4096 → 性能优势 ✓
```

### 不同缓冲区大小的性能对比（概念）

```
10MB 文件, 理论性能对比:

缓冲区大小 | 系统调用次数 | 相对 fgetc 的加速比
1 字节    | 20,000,000 | 1× (基准)
128 字节  | 163,840    | ~122×
512 字节  | 40,960     | ~488×
1024 字节 | 20,480     | ~977×
4096 字节 | 5,120      | ~3906×
8192 字节 | 2,560      | ~7812×
32768 字节| 640        | ~31250×

但实际加速比受磁盘速度限制，通常在 5-100× 之间。
缓冲区超过 128KB 后收益递减——磁盘带宽成为瓶颈。
```

### 常见错误（续）

| 错误 | 后果 | 正确 |
|------|------|---------|
| `fread` buf 是局部未初始化数组 | 读到随机数据 | `char buf[SIZE];` |
| 中文文件名用 `fopen` 默认编码 | Windows 下乱码 | 用 `_wfopen` (Win) 或 UTF-8 (Linux) |
| `fflush(stdin)` | 未定义行为 | fflush 只对输出流有意义 |
