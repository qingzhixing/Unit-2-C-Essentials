## Lesson 27 — 实现 my_memmove：内存重叠拷贝【重点】

### 课程任务

实现自己的 `memmove` 函数——`my_memmove(dest, src, n)`，将 `n` 字节从 `src` 拷贝到 `dest`，返回 `dest`。函数原型：

```c
void *my_memmove(void *dest, const void *src, size_t n);
```

**与 `memcpy` 的关键区别**：`memmove` 必须正确处理 `src` 和 `dest` 内存区域重叠的情况。这是本课的核心难点，也是技术面试中 C 语言内存模型理解的高频考点。

本题验证用例演示 `dest > src` 的经典场景——将字符串整体右移一位（这是 `memcpy` 无法正确处理的）：

```
输入 "abcd":  dest = buf+1, src = buf, n = 3  → 输出 "aabc"
输入 "hello": dest = buf+1, src = buf, n = 4  → 输出 "hhell"
输入 "xy":    dest = buf+1, src = buf, n = 1  → 输出 "xx"
```

### 为什么 memmove 是重点？【面试高频】

这是面试中的高频题，考察对 C 语言内存模型的深度理解。很多候选人能写出 `memcpy`，但当被问到"如果 src 和 dest 重叠会怎样"时，90% 的人答不上来。如果你能在白板上清晰地画出正向拷贝破坏数据的过程，并给出方向判断 + 反向拷贝的正确解法，你已经超越绝大多数候选人。

### 前置知识：三种内存关系全景

#### 情况 A：完全不重叠

```
                src                      dest
                ↓                        ↓
 mem: ... [a][b][c][d][\0] ... [ ][ ][ ][ ][ ] ...
            └──┬──┘              └──┬──┘
             重叠? 无 → 怎么拷贝都对 ✓
             memcpy ✅   memmove ✅
```

#### 情况 B：dest < src（目的在源前面）

```
            dest       src
            ↓          ↓
 mem: ... [ ][ ][ ][ ][a][b][c][d][\0] ...
            └────┬────┘
             重叠区间   └────┬────┘
              
 正向拷贝: 写入 dest 从左开始 → 离 src 还有距离 → 不会覆盖未读数据 ✓
 反向拷贝: 也安全 ✓
 memcpy:   ✅（虽然标准说不保证，但实际上正向拷贝可行）
 memmove:  ✅
```

#### 情况 C：dest > src（目的在源后面）← 本题测试场景！

```
            src        dest
            ↓          ↓
 mem: ... [a][b][c][d][ ][ ][ ][ ][\0] ...
            └────┬────┘
             src区间    └────┬────┘
                        重叠区间
             
 正向拷贝: 写入 dest 从左开始 → 立即覆盖了 src 的后半部分！ ✗
 反向拷贝: 从右端末尾开始 → 先读最后数据，再往前走 → 写入时已读完 ✓
 memcpy:   ❌（破坏数据）
 memmove:  ✅
```

### 核心原理：为什么正向拷贝会破坏数据？

这是本课最重要的部分。用逐字节跟踪的方式展示错误和正确两种做法。

#### 正向拷贝的灾难（错误！）

假设 `buf = "abcd\0"`，执行 `my_memmove(buf+1, buf, 3)`——将位置 0~2 的 3 个字节拷贝到位置 1~3。**正确结果应为 "aabc"**（第一个 'a' 不变，后面三个被 'a','b','c' 覆盖）。

```
初始状态（地址视角）:
 地址:  0    1    2    3    4
      ┌────┬────┬────┬────┬────┐
 buf: │ a  │ b  │ c  │ d  │ \0 │
      └────┴────┴────┴────┴────┘
        ↑         ↑
       src      dest
       (0)       (1)
       n = 3（拷贝 3 字节）

═══════════════════════════════════════════════════
正向拷贝逐步过程（每个 i 都标注了 buf 数组变化）：
═══════════════════════════════════════════════════

i=0: dest[0] = src[0]
     → 把 buf[1] 赋值为 buf[0] 的值 'a'
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ a  │ c  │ d  │ \0 │   ← 'b' 被覆盖了！
     └────┴────┴────┴────┴────┘
       ↑         ↑
      src      dest    但 src[0] 还没被碰到 ✓
      (0)       (1)

i=1: dest[1] = src[1]
     → 本应取 buf[1] 的值 'b'
     → 但 buf[1] 在上一轮被改成了 'a'！
     → 所以 src[1] = 'a'（错误！本该是 'b'）
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ a  │ a  │ d  │ \0 │   ← 'c' 被覆盖为 'a'
     └────┴────┴────┴────┴────┘
       ↑              ↑
      src           dest     src[2] 也被污染了！
      (0)            (2)

i=2: dest[2] = src[2]
     → buf[2] 现在是 'a'（被污染的值）
     → 所以 src[2] = 'a'（错误！本该是 'c'）
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ a  │ a  │ a  │ \0 │   ← 'd' 被覆盖为 'a'
     └────┴────┴────┴────┴────┘

结果: buf = "aaaa"  →  完全错误！应该是 "aabc" ❌
```

**根因分析**：正向拷贝先读 src、再写 dest。当 `dest > src` 时，写入的 dest 位置恰好是后续循环要读取的 src 位置。"读-写-读-写"的顺序导致了数据的连锁污染。

#### 反向拷贝的正确做法

```
初始状态（同上）:
      ┌────┬────┬────┬────┬────┐
 buf: │ a  │ b  │ c  │ d  │ \0 │
      └────┴────┴────┴────┴────┘
        ↑         ↑
       src      dest
       (0)       (1), n = 3

═══════════════════════════════════════════════════
反向拷贝逐步过程（从 i=n-1=2 开始，递减到 0）：
═══════════════════════════════════════════════════

i=2: dest[2] = src[2]
     → 把 buf[3] 赋值为 buf[2] 的值 'c'
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ b  │ c  │ c  │ \0 │   ← 'd' 被覆盖，但 'd' 不在拷贝范围内
     └────┴────┴────┴────┴────┘             不需要了，没问题 ✓

i=1: dest[1] = src[1]
     → 把 buf[2] 赋值为 buf[1] 的值 'b'
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ b  │ b  │ c  │ \0 │   ← 'c' 被覆盖，但原 'c' 已在 i=2 被拷贝
     └────┴────┴────┴────┴────┘     到安全位置 ✓

i=0: dest[0] = src[0]
     → 把 buf[1] 赋值为 buf[0] 的值 'a'
     
     ┌────┬────┬────┬────┬────┐
     │ a  │ a  │ b  │ c  │ \0 │   ← 'b' 被覆盖，但原 'b' 已在 i=1 被拷贝 ✓
     └────┴────┴────┴────┴────┘

结果: buf = "aabc" → 完全正确！✓
```

**根因分析**：反向拷贝从末尾（`i=n-1`）开始，**先读取最后需要的数据**，将其拷贝到安全位置；然后往前推，处理更靠前的数据。当后面的数据被覆盖时，它已经完成了迁移。这就是 memmove 的精髓——**拷贝方向由地址关系决定，而非一成不变**。

### 算法详解

#### 方向判断三步决策

```
            ┌─ dest < src ──→ 正向拷贝（和 memcpy 相同）
            │                      for (i=0; i<n; i++) d[i]=s[i];
指针比较 ──┼─ dest > src ──→ 反向拷贝
            │                      for (i=n; i>0; i--) d[i-1]=s[i-1];
            └─ dest == src ─→ 无需拷贝（直接返回）
```

#### 为什么反向拷贝必须用 `for (i=n; i>0; i--) d[i-1] = s[i-1]`？

```c
// ❌ 错误写法 1: 越界访问
for (i = n; i >= 0; i--) d[i] = s[i];   // s[n] 越界！最后一次访问 i=0 时 d[0]=s[0] 是对的
                                          // 但 i=n 时 d[n]=s[n] 访问了数组末尾之后的字节

// ❌ 错误写法 2: 少拷贝一个字节
for (i = n-1; i > 0; i--) d[i] = s[i];  // 漏掉了 i=0

// ✅ 正确写法: 两种等价形式
for (i = n; i > 0; i--) d[i-1] = s[i-1];   // 方式 1: 循环 n 次，用 i-1 索引
while (n-- > 0) d[n] = s[n];                 // 方式 2: 用递减的 n，最简洁
```

### memcpy vs memmove 完整对比

| 维度 | memcpy | memmove |
|------|--------|---------|
| 重叠处理 | 不保证（未定义行为 UB） | **保证**正确处理重叠 |
| 方向判断 | 无 | **有**（`if (d < s)` / `else if (d > s)`） |
| 性能 | 可能略快（无条件分支） | 多一个比较 + 条件跳转（通常可忽略） |
| 拷贝方向 | 总是正向 | dest<src:正向 / dest>src:反向 |
| 标准年份 | C89 | C89 |
| restrict | C99 起声明为 `restrict` | 无 `restrict` |
| 面试考点 | `void*` → `char*` 转换 | **内存重叠 + 方向判断** |
| 使用场景 | 拷贝到新分配 buffer | 数组内移动元素、不确定是否重叠 |
| 能否替代对方 | 不能替代 memmove | **可以替代 memcpy**（多一次判断） |

### 真实工程场景：什么时候必须用 memmove？

```c
// 场景 1: 数组中删除一个元素（后续元素前移）
int arr[] = {1, 2, 3, 4, 5};  // 删除 arr[1]=2
// 将 arr[2..4] 前移 → dest=arr+1, src=arr+2
// dest < src → 正向拷贝安全
memmove(arr+1, arr+2, 3 * sizeof(int));
// arr[] = {1, 3, 4, 5, 5}

// 场景 2: 数组中插入一个元素（后续元素后移）
int arr[10] = {1, 2, 3, 4, 5};  // 在位置 1 插入 99
// 先将 arr[1..4] 后移 → dest=arr+2, src=arr+1
// dest > src → 必须反向拷贝！memcpy 会出错
memmove(arr+2, arr+1, 4 * sizeof(int));
arr[1] = 99;
// arr[] = {1, 99, 2, 3, 4, 5}

// 场景 3: 字符串去除前缀
char *str = "   hello";  // 去掉前导空格
memmove(str, str+3, strlen(str+3)+1);  // dest < src → 正向安全
// str = "hello"
```

### 三个完整示例

**示例 1**：不重叠拷贝（与 memcpy 等价）
```
src = "hello\0" (0x100), dest = 新缓冲区 (0x200), n = 6
指针比较: 0x200 > 0x100 → dest > src → 反向拷贝
但不重叠！正向反向结果完全一样。
输出: "hello" ✓
```

**示例 2**：重叠拷贝 — dest < src（正向安全）
```
buf = "abcdef\0"
my_memmove(buf, buf+2, 3)  → 把 "cde" 拷贝到开头
dest=0, src=2 → dest < src → 正向拷贝 ✓
结果: buf = "cdedef\0", 打印前 3 字符 → "cde" ✓
```

**示例 3**：重叠拷贝 — dest > src（必须反向）← 本题
```
buf = "abcd\0"
my_memmove(buf+1, buf, 3)  → 把 "abc" 拷贝到位置 1~3
dest=1, src=0 → dest > src → 必须反向拷贝！
结果: buf = "aabc\0" → 打印 "aabc" ✓
```

### 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 不做方向判断，总是正向拷贝 | 重叠时数据被污染（连锁复制） | 检查 dest 与 src 的指针关系 |
| 方向判断写反（dest<src 时反向） | dest<src 反向拷贝也可能出错 | 画内存图确认 |
| 反向循环写成 `for(i=n; i>=0; i--)` | 访问 `s[n]` 越界 | `for(i=n; i>0; i--) d[i-1]=s[i-1]` |
| 方向判断用 `d < s` 但 d 和 s 指向不同对象 | C 标准未定义跨对象指针比较 | 只在同一数组/缓冲区中使用 |
| 忘记 `dest == src` 的情况 | 可能做无意义的拷贝或判断失败 | 三个分支: <, >, == |
| 只判断 `dest < src` 时正向，否则反向 | `dest == src` 时多一次无意义的反向拷贝 | 增加 `else if (dest > src)` 精确分支 |

### 重要知识点

- 指针比较（`d < s`）判断前后关系——这是 memmove 的核心
- 反向循环的两类写法：`while (n--)` 和 `for (i=n; i>0; i--)`
- 内存重叠的根因：写入覆盖了尚未读取的源数据
- `memmove` 是 `memcpy` 的安全超集——多一个方向判断，但可以处理所有场景
- C99 起 `memcpy` 声明为 `restrict`（暗示不重叠），`memmove` 无此限定

### 课堂讨论

1. **如果 dest 和 src 完全不重叠，正向和反向结果一样吗？** — 一样。但正向对 CPU 缓存预取更友好，非重叠场景可走正向分支。

2. **为什么 C 标准不直接把 memcpy 做成 memmove？** — 历史原因 + 性能。`memcpy` 假设程序员保证不重叠，省去判断。某些平台 memcpy 用 DMA/块传输指令优化，硬件要求 src/dest 不重叠。`memmove` 的 `if` 分支在绝大多数场景下开销可忽略，但 C 标准选择了更细粒度的控制。

3. **能用 memcpy 实现 memmove 吗？** — 部分可以。当 dest < src 时 memcpy 正向拷贝可行；当 dest > src 时必须反向拷贝，memcpy 做不到。所以答案是不能。

4. **指针比较 `d < s` 在所有 C 实现中都合法吗？** — C 标准保证：指向同一数组（或同一 `malloc` 分配的内存块）内的指针可以比较。但指向不同分配的对象的指针比较结果是未定义的。在 memmove 的实际使用中，src 和 dest 通常指向同一缓冲区或其副本，比较是安全且合法的。

5. **为什么本课的测试选择 `dest > src` 而不是 `dest < src`？** — 因为 `dest < src` 正向就能处理，体现不出 memmove 和 memcpy 的区别。`dest > src` 必须反向拷贝，是区分两者的唯一场景。这道题的设计就是为了让你踩坑——如果你用 memcpy 的正向拷贝，输出是 "aaaa"，而不是正确的 "aabc"。
