## Lesson 26 — 实现 my_memcpy：标准内存拷贝

### 课程任务

实现自己的 `memcpy` 函数——`my_memcpy(dest, src, n)`，将 `n` 字节从 `src` 逐字节拷贝到 `dest`，返回 `dest`。`swap` 辅助函数已提供（作参考）。函数原型与标准库一致：

```c
void *my_memcpy(void *dest, const void *src, size_t n);
```

- `dest`：目标缓冲区指针（可写）
- `src`：源缓冲区指针（只读，`const` 修饰）
- `n`：要拷贝的字节数（`size_t` 类型）
- 返回值：`dest`（目标缓冲区的起始地址）

### 前置知识：`void*` 为什么必须转 `char*`

#### `void*` — C 语言的"万能指针"

`void*` 可以接收任何类型的指针而无需显式强制转换：

```c
int    x = 42;
double y = 3.14;

void *p;
p = &x;    // ✅ int*   → void*
p = &y;    // ✅ double* → void*
```

**但 `void*` 有一个致命限制：不能做算术运算，不能解引用。**

```c
void *p = ...;
p++;       // ❌ 编译错误！void 无大小，编译器不知 +1 跳几个字节
p + 1;     // ❌ 编译错误！
*p;        // ❌ 编译错误！不能解引用（不知道读几个字节）
```

这就是 `memcpy` 必须先把 `void*` 转成 `char*` 的原因——只有知道了指向类型的大小，编译器才能生成正确的指针算术。

#### 为什么偏偏转 `char*`？能用 `int*` 吗？

因为 `char` 恰好是 **1 字节**，C 语言中最小的可寻址单位。`memcpy` 的定义就是"逐字节拷贝"。用 `int*`（4 字节）会导致：

```
假设拷贝 6 字节:

用 int*:   每次拷贝 4 字节 → 第一轮拷贝 4 字节 ✓, 第二轮剩余 2 字节
           → 但 int* += 1 会跳 4 字节，第二轮的 2 字节跨了两个 int → 越界！

用 char*:  每次拷贝 1 字节 → 循环 6 次，每次 1 字节 → 完美！✓
```

```
C 类型大小对照（64 位系统典型值）:
┌───────────┬──────────┬──────────────────────┐
│   类型    │   大小   │  ptr+1 跳过的字节    │
├───────────┼──────────┼──────────────────────┤
│   char    │  1 字节  │  1  ← 最适合逐字节   │
│   short   │  2 字节  │  2                    │
│   int     │  4 字节  │  4                    │
│   long    │  8 字节  │  8                    │
│   double  │  8 字节  │  8                    │
└───────────┴──────────┴──────────────────────┘
```

**`char*` vs `unsigned char*`**：标准规定 `memcpy` 行为等价于 `unsigned char` 的逐字节拷贝。用 `unsigned char*` 更严谨——避免有符号 char 在比较/赋值时的符号扩展问题。但本练习中 `char*` 够用，编译器优化后效果相同。

### 算法详解：逐字节搬运

#### 核心思路

```
1. 将 void* dest 强转为 char* d（可读写）
2. 将 const void* src 强转为 const char* s（只读）
3. 循环 n 次: d[i] = s[i]（逐字节赋值）
4. 返回 dest（保持原始指针语义）
```

#### 单步跟踪：拷贝 "hello" 的完整过程

```
输入: "hello\n"（stdin）→ fgets 读入 → src 指向 "hello\0..."
strlen(src) = 5, 调用 my_memcpy(dest, src, 6)

                                 初始状态:
    src 内存:                        dest 内存:
    ┌───┬───┬───┬───┬───┬───┐      ┌───┬───┬───┬───┬───┬───┐
    │ h │ e │ l │ l │ o │ \0│      │ ? │ ? │ ? │ ? │ ? │ ? │ ← 未初始化字节
    └───┴───┴───┴───┴───┴───┘      └───┴───┴───┴───┴───┴───┘
    ↑ s (0x100)                     ↑ d (0x200)
    const char *                    char *

逐字节跟踪:

| 循环 | 操作 | s[i] 值 | src 地址 | d[i] 之前 | d[i] 赋值后 | dest 变化 |
|-----|------|---------|----------|----------|------------|----------|
| i=0 | d[0]=s[0] | 'h' (0x68) | 0x100 | ? | 'h' | d[0]='h' |
| i=1 | d[1]=s[1] | 'e' (0x65) | 0x101 | ? | 'e' | d[1]='e' |
| i=2 | d[2]=s[2] | 'l' (0x6C) | 0x102 | ? | 'l' | d[2]='l' |
| i=3 | d[3]=s[3] | 'l' (0x6C) | 0x103 | ? | 'l' | d[3]='l' |
| i=4 | d[4]=s[4] | 'o' (0x6F) | 0x104 | ? | 'o' | d[4]='o' |
| i=5 | d[5]=s[5] | \0 (0x00)  | 0x105 | ? | \0 | d[5]=\0 |

拷贝后 dest:
┌───┬───┬───┬───┬───┬───┐
│ h │ e │ l │ l │ o │ \0│   ← dest 现在和 src 完全一致
└───┴───┴───┴───┴───┴───┘

my_memcpy 返回 dest (0x200)
main: printf("%s\n", dest) → 输出 "hello" ✓
```

### 类型转换详解

| 步骤 | 代码 | 含义 | 注意事项 |
|------|------|------|---------|
| 源指针转换 | `const char *s = (const char *)src;` | src 是 `const void*`，转换为 `const char*` | 必须保留 `const`！ |
| 目标指针转换 | `char *d = (char *)dest;` | dest 是 `void*`（可写），转为 `char*` | 不加 `const` |
| 逐字节赋值 | `d[i] = s[i];` | `char*` 的 `[i]` 是第 i 个字节 | 等价于 `*(d+i) = *(s+i)` |
| 返回 | `return dest;` | 返回原始指针，不是 `d` | 与标准库行为一致 |

**为什么返回 `dest` 而不是 `d`？** 语义上两者值相同（都指向同一地址），但返回 `dest` 更清晰地表达"返回目标缓冲区起点"的意图，与标准库行为一致。

**为什么 `s` 必须保留 `const`？**

```c
// ❌ 错误：丢掉 const 限定符
char *s = (char *)src;         // 编译器警告: discards 'const' qualifier
                               // 而且允许 s 被用来修改本该只读的 src 数据

// ✅ 正确：保留 const
const char *s = (const char *)src;   // 编译器无警告，src 数据受保护
```

`src` 声明为 `const void*`，表示调用者承诺"请勿修改我提供的数据"。丢掉 `const` 破坏了这一承诺。

### memcpy vs 逐元素赋值 vs strcpy

| 维度 | memcpy | 逐元素 `a[i]=b[i]` | strcpy |
|------|--------|-------------------|--------|
| 适用类型 | 任意（通过 `void*`） | 需要知道具体类型 | 只适用字符串 |
| 操作粒度 | 字节级 | 元素级 | 字节级，遇 `\0` 停止 |
| 长度指定 | `n` 字节（显式） | 循环次数（显式） | 自动（遇 `\0` 停止） |
| 泛型能力 | ✅ 一份代码处理所有类型 | ❌ 每种类型独立循环 | ❌ 只处理 char |
| 安全性 | 需确保 n 不越界 | 需确保 n 不越界 | 需确保 dest 够大 |
| `\0` 拷贝 | 包含（若 n 包含） | 包含（若 n 包含） | 包含（自动） |

### 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 忘记 `void*`→`char*` 转换 | 编译错误 | `char *d = (char *)dest;` |
| `src` 转换丢掉 `const` | 编译器警告，const 保护失效 | `const char *s = (const char *)src;` |
| 用 `int*` 代替 `char*` | 只能拷贝 4 的倍数字节，余数丢失 | 用 `char*` |
| 返回 `d` 而非 `dest` | 功能正确但语义不清晰 | `return dest;` |
| 循环条件用 `i < n` 但 `i` 是 `int`，`n` 是 `size_t` | 有符号/无符号比较警告 | `size_t i; for (i=0; i<n; i++)` |

### 重要知识点

- `void*` → `char*` 转换是操作任意二进制内存的基础技巧
- `size_t` 是无符号整型（`<stddef.h>` 定义，`<string.h>` 包含）
- `const` 限定符必须在指针转换中保留
- 逐字节循环是 `memcpy` 的语义本质——即使标准库用 SIMD/块传输优化，语义等价
- `memcpy` 不检查内存重叠——调用者需自行保证 `src` 和 `dest` 不重叠

### 课堂讨论

1. **为什么参数顺序是 `(dest, src, n)` 而不是 `(src, dest, n)`？** — 类比赋值语句 `dest = src`，目的在左、源在右。这是 UNIX 哲学的一致性体现。

2. **如果 `n=0`，memcpy 应该做什么？** — 不拷贝任何字节，直接返回 `dest`。循环条件 `i < 0` 不执行。标准保证此行为安全。

3. **`size_t` 和 `int` 有什么区别？为什么 memcpy 用 `size_t`？** — `size_t` 是无符号类型，保证能表示任何对象的大小。`int` 有符号且通常 32 位（最大约 2GB），拷贝超过 2GB 数据时会溢出。

4. **memcpy 能用来拷贝结构体吗？** — 可以，但要区分浅拷贝和深拷贝。对简单结构体（POD，Plain Old Data），`memcpy` 等效逐字段赋值（浅拷贝）。含指针成员的结构体，`memcpy` 只拷贝指针值，两个结构体会指向同一内存——可能产生悬挂指针。此时需要深拷贝（对指针指向的内容单独拷贝）。

5. **为什么 C 标准要提供 `memcpy` 和 `memmove` 两个函数？** — `memcpy` 假设不重叠，可以无脑正向拷贝（编译器可能优化为块指令）。`memmove` 保证重叠正确处理，但多一个方向判断。这是经典的"C 语言给予程序员选择权"：你知道不重叠，就可以用更快的 `memcpy`。
